<DocLink bold={false} product={props.product} path='user-guide/alarms' target="_blank">Alarms</DocLink> are notifications generated when monitored values cross defined thresholds. In this lesson you will configure alarm rules on the `air-sensor` device profile, add an Alarms table widget to the air sensor state, test triggering, and propagate the widget across every dashboard state so alarms are visible at every level of the building hierarchy.

---

## Step 4.1 Configure Alarm Rules

<DocLink bold={false} product={props.product} path='user-guide/device-profiles#alarm-rules' target="_blank">Alarm rules</DocLink> are defined at the **device profile** level and apply automatically to every device of that profile. Each rule can have multiple create conditions with different severities, and a clear condition that resolves the alarm automatically when the value returns to a safe range.

### High Temperature

<Steps>
1. Go to the **Alarm rules** page and click **+ &#8702; Create new alarm rule**.
2. In the **General** section set:
   - **Alarm type:** `High temperature`
   - **Entity type:** Device profile
   - **Device profile:** `air-sensor`
3. In the **Arguments** section click **Add argument** and fill in:
   - **Entity type:** Current entity
   - **Argument type:** Latest telemetry
   - **Time series key:** `temperature`
   - **Argument name:** `temperature`
   Click **Add**.
4. In **Create conditions** click **Add create condition**:
   - **Severity:** Critical
   - Click **Add condition &#8702; Add argument filter**:
     - Argument: `temperature`, Value type: Numeric
     - Filter: operation **greater than**, value `26`
   Click **Add**, then **Save**.
5. Click **Add create condition** again:
   - **Severity:** Major
   - Click **Add condition &#8702; Add argument filter**:
     - Argument: `temperature`, Value type: Numeric
     - Filter 1: **greater than** `24`
     - Filter 2: **less or equal** `26`
   Click **Add**, then **Save**.
6. In **Clear conditions** click **Add clearing condition**:
   - Click **Add condition &#8702; Add argument filter**:
     - Argument: `temperature`, Value type: Numeric
     - Filter: **less or equal** `24`
   Click **Add**, then **Save**.
7. Expand **Advanced settings**, enable **Propagate alarm to related entities**, set **Relation type** to `Contains`.
8. Click **Add** to save the rule.
</Steps>

<Aside type="tip">
  **Propagation** means that when an alarm fires on a device, it automatically appears on every related entity up the hierarchy — the office it belongs to, and the building that office belongs to. This makes it possible to monitor alarms at the building level without querying each device directly.
</Aside>

### Additional Alarm Rules

Create the following rules for the `air-sensor` profile using the same steps. Enable **Propagate alarm to related entities** (Relation type: `Contains`) for each.

| Alarm type | Major condition | Critical condition | Clear condition |
|------------|-----------------|--------------------|-----------------|
| Low temperature | `temperature < 20` AND `temperature ≥ 18` | `temperature < 18` | `temperature ≥ 20` |
| High humidity | `humidity > 60` AND `humidity ≤ 65` | `humidity > 65` | `humidity ≤ 60` |
| Low humidity | `humidity < 40` AND `humidity ≥ 35` | `humidity < 35` | `humidity ≥ 40` |
| High CO₂ | `co2 ≥ 490` AND `co2 < 500` | `co2 > 500` | `co2 < 490` |

## Step 4.2 Add Alarm Widget to Air Sensor State

The **Alarms table** widget displays active alarms for the entity currently selected in the dashboard state. Add it to the `air_sensor` state first, then propagate it to the other states in the following steps.

<Steps>
1. Click the **SD-001** row in the **Office sensors list** to navigate to the **air_sensor** state.
2. Enter edit mode and click **+ Add widget**.
3. Find the **Alarm widgets** bundle and select the **Alarms table** widget.
4. Set the datasource to the **Selected entity** alias.
5. In the **Filter** section, check **Active** in the alarm status list (show only active alarms).
6. Remove the **Assignee** column from the columns list.
7. Set the widget title to **Air sensor alarms**.
8. In **Table buttons**, leave only **Allow alarms acknowledgment** and **Allow alarms clear** enabled.
9. Disable **Data export** in the Card appearance section, then click **Add**.
10. Resize and place the widget on the dashboard, then save.
</Steps>

## Step 4.3 Test Alarm Triggering

Verify the alarm rule works by manually sending a temperature value above the threshold via the device connectivity tool — no need to wait for the telemetry generator.

<Steps>
1. Go to **Entities &#8702; Devices** and open **SD-001** (Indoor Air Quality Sensor).
2. Click **Check connectivity** in the Device details panel.
3. Select your messaging protocol and operating system. Copy the provided `curl` command.
4. In the command, change the temperature value to `25` (triggers Major) or `27` (triggers Critical).
5. Paste and execute the command in your terminal.
6. Return to the dashboard — the alarm should appear in the **Air sensor alarms** widget within a few seconds.
</Steps>

### Acknowledge an Alarm

2. Confirm the action in the dialog.

<Steps>

## Step 4.4 Propagate Alarm Widget to All States

Copy the Alarms table widget to each remaining state and re-point its datasource to the alias that represents entities at that level. By enabling **Search propagated alarms**, the widget at a higher level (office, building) will show alarms from all child entities — not just the entity directly selected.

### Office State

<Steps>
1. In the **air_sensor** state (edit mode), right-click the **Air sensor alarms** widget and select **Copy**.
2. Navigate to the **office** state, right-click an empty area, and select **Paste**.
3. Place and resize the widget.
4. Click the **pencil** icon on the pasted widget to edit it.
5. Change the alarm source alias to **Office sensors** (the Device search query alias created in Lesson 2).
6. Enable **Search propagated alarms**.
7. Change the widget title to **Office alarms**.
8. Click **Apply**, then save the dashboard.
</Steps>
