**Device profiles** let you centrally define and manage common settings for a group of similar devices — transport protocol, rule chain, alarm rules, calculated fields, firmware, and provisioning strategy. Instead of configuring each device individually, you define the profile once and assign it to as many devices as needed.

## Managing Profiles

### Create a Profile

<Steps>
1. Navigate to **Profiles &#8702; Device profiles**.
2. Click **+ Add device profiles** and select **Create new device profile**.
3. Enter a **Name** (the only required field).
4. Optionally configure the transport type, rule chain, queue, alarm rules, calculated fields, and provisioning.
5. Click **Add**.
</Steps>

<Aside type="tip">
  **Example:** a `Thermometer` profile using MQTT transport, with a Critical alarm rule for temperature above 40 °C, assigned to every temperature sensor in the tenant — one configuration, applied to all of them at once.
</Aside>

### Edit a Profile

<Steps>
1. Click the profile name or the **pencil** icon.
2. Update the desired settings across any tab.
3. Click **Apply changes**.
</Steps>

### Delete a Profile

Click the **trash** icon next to the profile and confirm deletion.

## Key Settings

### Transport Configuration

The **Transport type** dropdown determines how devices in this profile communicate with ThingsBoard: **Default** (MQTT, HTTP, or CoAP with standard ThingsBoard payloads), **MQTT** (custom topics and payload format), **CoAP**, **LWM2M**, or **SNMP**. Changing it doesn't migrate existing device credentials — devices already using incompatible credentials will need new ones for the new transport.

See <DocLink product={props.product} path='user-guide/connectivity-guide/#direct-connection' target="_blank">How to Connect IoT Devices</DocLink> for a full comparison of protocols and links to each one's API reference.

### Default Rule Chain

By default, the Root Rule Chain processes all incoming messages and events for every device. As your system grows and you introduce different device types, the Root Rule Chain can become complex.

Assign a dedicated <DocLink product={props.product} path='user-guide/rule-engine/#rule-chain' target="_blank">rule chain</DocLink> to a device profile to isolate processing logic — telemetry handling, lifecycle events, and state tracking — for that device type without overloading the Root Rule Chain.

### Default Queue

By default, all messages use the **main** queue. Assign a custom queue to isolate high-priority devices (e.g., fire alarms) from high-volume ones (e.g., water meters), or to apply different submit and processing strategies. See <DocLink product={props.product} path='user-guide/rule-engine/queues' target="_blank">Queues</DocLink> for available delivery guarantees and configuration.

<Aside type="note">
  Custom queues must be configured by a System Administrator before they can be assigned to a profile.
</Aside>

### Alarm Rules

Define conditions that apply to all devices in this profile — for example, trigger a Critical alarm when temperature exceeds 40 °C. Rules defined here are shared across every device using the profile. See <DocLink product={props.product} path='user-guide/alarm-rules' target="_blank">Alarm Rules</DocLink>.

### Calculated Fields

Define formulas that compute derived metrics from telemetry or attributes — for example, converting a raw sensor reading to a different unit. Like Alarm Rules, calculated fields defined here are inherited by every device using the profile; a device-level calculated field runs alongside any profile-level one, not instead of it. See <DocLink product={props.product} path='user-guide/calculated-fields' target="_blank">Calculated Fields</DocLink>.

### Firmware and Software

Assign a firmware or software package to roll it out to every device on this profile at once — a per-device assignment overrides the profile assignment for that device only. See <DocLink product={props.product} path='user-guide/ota-updates/#to-a-device-profile' target="_blank">OTA Updates</DocLink> for the full update flow and rollout controls.

### Provisioning

Control how new devices are created and authenticated when they first connect. See <DocLink product={props.product} path='user-guide/provisioning/#configure-in-device-profile' target="_blank">Configure Provisioning in a Device Profile</DocLink>.

### Mobile Dashboard

The dashboard the mobile app uses as the device details screen for devices on this profile. Unrelated to the per-user login dashboard covered in <DocLink product={props.product} path='user-guide/dashboards' target="_blank">Dashboards</DocLink>.

### Default Edge Rule Chain

Only relevant if the tenant also runs ThingsBoard Edge; it's the rule chain Edge instances use to process incoming data for devices on this profile.

## Profile Changes and Existing Devices

Editing a profile's **Rule Chain**, **Queue**, **Alarm Rules**, or **Calculated Fields** immediately affects every device already assigned to that profile — there's no need to touch individual devices. This is the main advantage of configuring these settings at the profile level instead of per device.

<Aside type="caution">
  Changing the **transport type** does not update existing device credentials. A device with credentials for the old transport won't be able to connect until it's issued new, compatible credentials — see <DocLink product={props.product} path='user-guide/devices/#manage-credentials' target="_blank">Manage Credentials</DocLink>.
</Aside>

Moving a device to a *different* profile (in <DocLink product={props.product} path='user-guide/devices/#edit-device' target="_blank">Edit Device</DocLink>) applies that profile's rule chain, queue, alarm rules, and calculated fields going forward — it does not retroactively change alarms or calculated field results already generated under the previous profile.

## Troubleshooting

| Symptom | Cause | Fix |
|---------|-------|-----|
| Device can't connect after switching its profile | New profile uses a different transport type than the device's current credentials support | Issue new, compatible credentials — see <DocLink product={props.product} path='user-guide/devices/#manage-credentials' target="_blank">Manage Credentials</DocLink> |
| Alarm rule or calculated field doesn't seem to run | It was added on the device instead of the profile (or vice versa) | Confirm which level the rule was created at — profile-level rules apply to every device on that profile, device-level rules apply only to that one device |
| Custom queue isn't selectable when editing a profile | The queue hasn't been created yet | Ask a System Administrator to configure the queue first — see <DocLink product={props.product} path='user-guide/rule-engine/queues' target="_blank">Queues</DocLink> |
| Firmware assigned to a profile doesn't reach a specific device | That device has its own direct firmware assignment, which overrides the profile | Check the device's individual firmware assignment — see <DocLink product={props.product} path='user-guide/ota-updates' target="_blank">OTA Updates</DocLink> |

## Where Device Profiles Are Used

- <DocLink product={props.product} path='user-guide/devices' target="_blank">Devices</DocLink> — every device is assigned exactly one profile, inheriting its transport, rule chain, and alarm/calculated field configuration
- <DocLink product={props.product} path='user-guide/ota-updates' target="_blank">OTA Updates</DocLink> — assign firmware or software to every device on a profile at once
- <DocLink product={props.product} path='user-guide/rule-engine' target="_blank">Rule Engine</DocLink> — the profile's rule chain and queue route and process telemetry from all its devices
- <DocLink product={props.product} path='user-guide/alarm-rules' target="_blank">Alarm Rules</DocLink> and <DocLink product={props.product} path='user-guide/calculated-fields' target="_blank">Calculated Fields</DocLink> — shared logic applied across every device using the profile
- <DocLink product={props.product} path='reference/rest-api' target="_blank">REST API</DocLink> — query, create, and manage device profiles programmatically
