Each integration type corresponds to a specific external system — a protocol gateway, IoT cloud platform, LoRaWAN network server, or message broker. You choose the type based on where your device data is coming from, not what the data contains.

## Standard protocols

Devices that speak a standard protocol directly, no intermediate platform.

<PlatformIntegrationsGrouped product={props.product} group="standard-protocols" />

## LoRaWAN network servers

LoRaWAN sensors forwarding messages through a network server.

<PlatformIntegrationsGrouped product={props.product} group="lorawan-servers" />

## Cloud IoT platforms

Devices already managed on a cloud IoT platform.

<PlatformIntegrationsGrouped product={props.product} group="cloud-iot" />

## Message brokers & queues

Data pipelines, event streams, or existing message infrastructure.

<PlatformIntegrationsGrouped product={props.product} group="message-brokers" />

## Industrial / OPC

PLCs and industrial equipment exposed through an OPC-UA server.

<PlatformIntegrationsGrouped product={props.product} group="industrial-opc" />

## NB-IoT / Cellular

Low-power wide-area devices on cellular IoT networks.

<PlatformIntegrationsGrouped product={props.product} group="nb-iot-cellular" />

## Consumer IoT

Smart home and consumer devices on the Tuya cloud.

<PlatformIntegrationsGrouped product={props.product} group="consumer-iot" />

## Custom

Any other source with a custom payload format.

<PlatformIntegrationsGrouped product={props.product} group="custom" />


<Aside type="note">
  **TCP**, **UDP**, and **Custom** integrations run as **remote-only** — they must be deployed outside the main ThingsBoard process. All other types support both embedded and remote deployment.
</Aside>
