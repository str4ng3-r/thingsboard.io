**Devices** are the primary IoT entities in ThingsBoard. They represent physical or virtual things — sensors, controllers, gateways, emulators — that publish telemetry, report attributes, and respond to <DocLink product={props.product} path='user-guide/command-and-control' target="_blank">RPC commands</DocLink>.

## Managing Devices

### Create Device

<Steps>
1. Navigate to **Entities &#8702; Devices**.
2. Click **+ Add device** and select **Add new device**.
3. Enter a **Name** (must be unique within the tenant).
4. Optionally set a <DocLink product={props.product} path='user-guide/device-profiles' target="_blank">Device profile</DocLink> (defaults to `default`), **Label**, or **Description**.
5. If this device will proxy data for other devices, enable **Is gateway** (see **[Gateway Devices](#gateway-devices)**).
6. Click **Add**.
</Steps>

{props.product !== Products.CE && (
<Aside type="tip">
  The creation form also includes <strong>Owner</strong> (Tenant or a specific Customer) and <strong>Groups</strong> fields for organization and access control.
</Aside>
)}

A connectivity check dialog opens automatically right after creation — continue below to verify the device and send its first data point.

### Check Connectivity

Verify that the device can communicate with ThingsBoard:

<Steps>
1. Open the device details page and click **Check connectivity** (or use the dialog that opened automatically after creation).
2. Select the protocol (MQTT, HTTP, or CoAP) and your operating system.
3. Copy the generated command and run it in your terminal.
</Steps>

On success, the device status changes from **Inactive &#8702; Active** and telemetry appears under the **Latest telemetry** tab.

<Aside type="tip">
  Choosing between a raw MQTT/HTTP/CoAP command, an IoT Gateway, or a platform integration? See <DocLink product={props.product} path='user-guide/connectivity-guide' target="_blank">How to Connect IoT Devices</DocLink> for the full comparison.
</Aside>

Having trouble? See **[Troubleshooting](#troubleshooting)** below.

### Generate Dashboard with AI

<Banner variant="peFeature" product={props.product}>Generate dashboard with AI is available in <DocLink product={Products.PE} path="installation/" target="_blank">ThingsBoard Professional</DocLink> and <a href="/installations/?product=thingsboard-cloud" target="_blank" rel="noopener noreferrer" class="tb-banner-ext"><b>ThingsBoard Cloud</b></a> only.</Banner>

**Generate dashboard with AI** builds a ready-to-use dashboard for this device in a single click — no prompt needed. AI reviews the device's available telemetry and attributes and automatically selects the widgets that best fit that data — value cards, trend charts, gauges, an alarms table, and more.

<Aside type="note">
  Not the same as <DocLink product={props.product} path="user-guide/dashboards/#configure-dashboard-with-ai" target="_blank">Configure Dashboard with AI</DocLink> on the Dashboards page — that one builds or edits a dashboard from a written description and can span multiple entities. **Generate dashboard with AI** is a single click, scoped to this device, and needs no prompt.
</Aside>

You can trigger it from two places:
- The connectivity check dialog — opens automatically after creating a device, or reopen it anytime via **[Check Connectivity](#check-connectivity)**.
- The device details panel, among its action buttons (see **[Device Details](#device-details)**).

<Aside type="tip">
  The device needs telemetry data before a dashboard can be generated — sent by the device itself, or via the sample command from the connectivity check dialog.
</Aside>

<Steps>
1. Click **Generate dashboard with AI**.
2. Wait about a minute while AI reads the device's data and analyzes its telemetry and attributes.
3. The dashboard opens automatically, named after the device (e.g., **SN-001 Dashboard**), populated with widgets matching its data — including an alarms widget if the device has active alarms.
</Steps>

Prefer to build it yourself? Create a blank <DocLink product={props.product} path="user-guide/dashboards" target="_blank">dashboard</DocLink> and add widgets bound to this device through an entity alias.

### Edit Device

<Steps>
1. Click the **pencil** icon next to the device in the list.
2. Update the name, profile, label, firmware, or software version.
3. Click **Apply changes**.
</Steps>

### Delete Device

Click the **trash** icon next to the device and confirm, or open the device details page and click **Delete device**.

### Manage Credentials

<Steps>
1. Open the device details page.
2. Click **Manage credentials**.
3. Select the credential type and fill in the required fields.
4. Click **Save**.
</Steps>

<Aside type="tip">
  Credential types (Access Token, X.509 Certificate, MQTT Basic) depend on the transport protocol used by your device. See <DocLink product={props.product} path='user-guide/connectivity-guide'>How to Connect IoT Devices</DocLink> for a full list of supported protocols and their credential requirements.
</Aside>

## Gateway Devices

A gateway device proxies data for multiple downstream devices over a single connection using the <DocLink product={props.product} path='reference/gateway-api/overview' target="_blank">Gateway API</DocLink>. Enable gateway mode when creating or editing a device.

| Setting | Description |
|---------|-------------|
| **Is gateway** | Grants the device access to the gateway topic namespace for proxying downstream devices. |
| **Overwrite activity time** | When enabled, the gateway's own connection activity keeps connected downstream devices in **Active** state. Useful when downstream devices are connected through the gateway but not actively sending telemetry — RPC commands and shared attribute updates can still be delivered to them. |

To enable gateway mode, toggle **Is gateway** during **[device creation](#create-device)** or in the device edit form.

<Aside type="note">
  Enabling **Is gateway** only grants a device access to the gateway topic namespace — it doesn't install anything. If you need a ready-made bridge for protocols like Modbus, OPC-UA, or BLE, use the <DocLink product={props.product} path='user-guide/connectivity-guide/#iot-gateway' target="_blank">ThingsBoard IoT Gateway</DocLink> instead, which uses this same API under the hood.
</Aside>

## Device Details

Clicking a device opens its details panel. The header exposes the following actions:

| Action | Description |

| **Manage credentials** | Change how the device authenticates — see **[Manage Credentials](#manage-credentials)**. |
| **Manage owner and groups** | Change the device's owner and group membership (PE/Cloud only). |

| **Delete device** | Deletes the device — see **[Delete Device](#delete-device)**. |

## Troubleshooting

| Symptom | Cause | Fix |
|---------|-------|-----|
| Device stays **Inactive** after running the connectivity command | Access token is wrong, expired, or belongs to another device | Copy the token again from **[Manage Credentials](#manage-credentials)** and re-run the command |
| Device stays **Inactive** after running the connectivity command | The command's protocol doesn't match the device | Confirm the device's <DocLink product={props.product} path="user-guide/device-profiles" target="_blank">Device profile</DocLink> uses the same transport (MQTT, HTTP, or CoAP) as the command |
| Command fails to connect or times out | Wrong host/port, or the port is blocked | Confirm the host in the generated command matches your ThingsBoard instance, and that outbound traffic on that port isn't blocked by a firewall or proxy |
| Command succeeds but no telemetry appears | Payload isn't valid JSON, or uses an unexpected key | Check the terminal output for an HTTP error status or broker rejection, and validate the payload |
| **Generate dashboard with AI** shows "No telemetry data found" | The device hasn't sent any data yet | Send at least one telemetry data point (see **[Check Connectivity](#check-connectivity)**), then try again |

## Where Devices Are Used

- <DocLink product={props.product} path='user-guide/dashboards' target="_blank">Dashboards</DocLink> — visualize telemetry, attributes, and alarms via widgets
- <DocLink product={props.product} path='user-guide/rule-engine' target="_blank">Rule Engine</DocLink> — process incoming telemetry, trigger alarms, route messages
- <DocLink product={props.product} path='user-guide/calculated-fields' target="_blank">Calculated Fields</DocLink> — compute derived values from device data
- <DocLink product={props.product} path='user-guide/digital-twins/relations' target="_blank">Relations</DocLink> — model hierarchies (e.g., Gateway → Contains → Sensor)
- <DocLink product={props.product} path='user-guide/integrations' target="_blank">Integrations</DocLink> — ingest data from third-party platforms and protocols instead of connecting devices directly
- <DocLink product={props.product} path='user-guide/command-and-control' target="_blank">Command & Control</DocLink> — send RPC commands to devices and receive their responses
- <DocLink product={props.product} path='reference/rest-api' target="_blank">REST API</DocLink> — query, create, and manage devices programmatically
