**Provisioning** is how devices get registered in ThingsBoard with credentials before they start sending data. There are two independent approaches: **Bulk Provisioning** (an admin imports a CSV file) and **Auto Provisioning** (devices self-register on first boot). Both result in the device existing in ThingsBoard with credentials ready to use.

This is distinct from <DocLink product={props.product} path='user-guide/claiming' target="_blank">Claiming</DocLink>: provisioning creates the device and gives it credentials; claiming transfers an already-provisioned device to a specific customer.

## Bulk Provisioning

**Bulk provisioning** imports a list of devices or assets into ThingsBoard from a CSV file in one operation. Each row creates (or updates) one entity and can include attributes, telemetry, and device credentials тАФ no API scripting required.

A common use case is pre-loading a batch of devices before shipping so they can use <DocLink product={props.product} path='user-guide/provisioning#auto-provisioning'>Auto Provisioning</DocLink> with the **Check Pre-Provisioned Devices** strategy: the devices are known to ThingsBoard in advance and only those exact devices are allowed to self-register.

### What You Can Import

| Column type | Description |
|-------------|-------------|
| **Name** | Entity name (required) |
| **Type** | Device or Asset type |
| **Label** | Optional display label |
| **Server attribute** | Any key тАФ stored as a server-side attribute |
| **Shared attribute** | Any key тАФ stored as a shared attribute (devices can read) |
| **Telemetry** | Any key тАФ stored as a time-series data point |
| **Access token** | Device credential тАФ Access Token value |
| **X.509** | Device credential тАФ X.509 certificate (PEM-encoded value) |
| **MQTT client ID** | Device credential тАФ MQTT Basic: client ID |
| **MQTT username** | Device credential тАФ MQTT Basic: username |
| **MQTT password** | Device credential тАФ MQTT Basic: password |

<Aside type="note">
  Name, Type, and all credential columns can each be mapped to only one column per import. Each device supports only one credential type тАФ do not mix Access Token, MQTT Basic, and X.509 columns in the same import. If multiple credential types are mapped, the system applies priority order: X.509 &#8594; MQTT Basic &#8594; Access Token.
</Aside>

### Import Process

<Steps>
1. Navigate to **Entities &#8702; Devices** (or **Assets**) and click the **Import** icon.
2. Upload your CSV file. Set the delimiter and toggle **First line contains column names** if applicable.
3. Enable **Update attributes/telemetry** if you want matching existing entities to be updated rather than flagged as errors.
4. Map each CSV column to a ThingsBoard field type (Name, Type, attribute, telemetry, Access Token, etc.).
5. Review the results тАФ the summary shows how many entities were created, updated, or failed.
</Steps>

### CSV Format

The file must have at minimum a name column and a type column. All other columns are optional. Empty cells are allowed тАФ missing values are simply skipped for that entity.

```csv
name,type,label,serialNumber,accessToken
Sensor-001,Temperature Sensor,Warehouse A,SN-4A21F,tok-a1b2c3d4
Sensor-002,Temperature Sensor,Warehouse B,SN-4A22F,tok-e5f6g7h8
Gateway-001,IoT Gateway,Main Hub,GW-0010A,tok-i9j0k1l2
```

### Bulk Provisioning via REST API

For automated pipelines тАФ CI/CD, ERP integrations, or manufacturing systems тАФ the same import can be triggered programmatically. See <DocLink product={props.product} path='reference/rest-api' target="_blank">Bulk Import REST API</DocLink> for the endpoint and request format.

## Auto Provisioning

Devices self-register on first boot by sending a provisioning request with a key and secret embedded in firmware. ThingsBoard validates it, creates or finds the device, and returns the credentials the device will use for all subsequent communication. No admin action needed per device.

<ImageGallery images={[{ src: '/src/assets/schemas/auto-provisioning.svg', alt: 'Auto provisioning diagram', caption: 'On first boot, the device sends a provisioning request with key and secret. ThingsBoard validates, creates or finds the device, and returns credentials for normal operation.' }]} />

Once provisioned, the device's `provisionState` server-side attribute is set to `provisioned`. Re-sending a provisioning request after that will be rejected with a failure response тАФ provisioning is a one-shot operation.

To re-provision a device, the reset procedure depends on the strategy:

**Allow Creating New Devices** тАФ delete the device in ThingsBoard. The device can then provision again as a new entity.

**Check Pre-Provisioned Devices** тАФ delete the `provisionState` server-side attribute on the device (**device &#8702; Attributes &#8702; Server attributes**). The device name and credentials remain intact. Via REST API: `DELETE /api/plugins/telemetry/DEVICE/{deviceId}/SERVER_SCOPE?keys=provisionState`

### Strategies

**Allow creating new devices** тАФ ThingsBoard creates the device on first request. Use this when device names (e.g. MAC addresses) aren't known at manufacturing time but are readable by firmware at runtime.

**Check Pre-Provisioned Devices** тАФ ThingsBoard only accepts devices already present in the system and rejects all others. Use bulk provisioning to load the approved list first. Good for regulated or high-security deployments where unknown devices must be blocked.

### Configure in Device Profile

In the <DocLink product={props.product} path='user-guide/device-profiles' target="_blank">Device Profile</DocLink>, go to the **Device Provisioning** tab, select a strategy, and copy the generated **Provision device key** and **Provision device secret** тАФ these go into the device firmware.

<Aside type="note">
  Each device profile has its own key/secret pair. All devices using the same profile share the same provisioning credentials but receive unique device credentials in return.
</Aside>

<Aside type="caution">
  The credential type is **not** configured here. The Device Provisioning tab only sets the strategy and the key/secret pair тАФ how the credential type is determined is described in [Credential Types](#credential-types) below.
</Aside>

### Credential Types

How the credential type is determined depends on the strategy:

**Allow Creating New Devices** тАФ the device specifies the type in the provision request body (`credentialsType`: `ACCESS_TOKEN`, `MQTT_BASIC`, or `X509_CERTIFICATE`). ThingsBoard creates the device with exactly those credentials.

**Check Pre-Provisioned Devices** тАФ ThingsBoard returns whatever credentials are already configured on the device, regardless of what the provision request body contains. To use a specific credential type (for example, MQTT Basic), set it on the device before it provisions тАФ via the device **Credentials** tab in the UI, or via `POST /api/device-with-credentials` for bulk setup. Values sent in the provision request body are ignored.

| Type | How it works | Best for |
|------|-------------|----------|
| **Access Token** | ThingsBoard generates a unique token | Simplest setup, most protocols |
| **MQTT Basic** | Device supplies username, password, and client ID | MQTT devices that manage their own identity |
| **X.509 Certificate** | Device provides a certificate; ThingsBoard returns credentials ID | Highest security; requires MQTT over SSL |
