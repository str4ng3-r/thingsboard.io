**Device claiming** is how a device provisioned by a tenant gets transferred to a specific customer. The tenant or device provides a secret key; the customer enters that key to take ownership. Once claimed, only that customer's users can access the device.

## Choose a Scenario

| | Device-Side | Server-Side |
|-|-------------|-------------|
| **Who generates the secret** | The device itself, at runtime | The tenant admin, in advance |
| **How the customer gets the key** | Device displays it (screen, QR code, LED blink) | Admin distributes it out-of-band (email, box label) |
| **Best for** | Wi-Fi / IP devices with a display or button | NB-IoT / LoRaWAN / Sigfox devices without a UI |
| **Key stored in** | ThingsBoard (sent by device via transport API) | `claimingData` server-side attribute |

Both scenarios end the same way: the customer opens the **Claim Device** widget, enters the device name and secret key, and ownership transfers.

## Device-Side Claiming

The device generates a secret key on a trigger (first boot, button press) and publishes it to ThingsBoard. ThingsBoard stores it temporarily. The customer reads the key from the device and claims it.

### 1. Enable Claiming

Claiming must be enabled before a device can publish claiming data. Choose one:

- **Globally** — set `SECURITY_CLAIM_ALLOW_CLAIMING_BY_DEFAULT=true`
- **Per device** — add a server-side attribute `claimingAllowed = true` on the device

### 2. Device Publishes the Claiming Key

The device sends the following payload over any supported transport (MQTT, HTTP, CoAP):

```json
{
  "secretKey": "your-secret",
  "durationMs": 60000
}
```

- `secretKey` — the secret customers must provide. Omit to use an empty string.
- `durationMs` — how long the key is valid (milliseconds). Omit to use the system default (`SECURITY_CLAIM_DURATION`). Maximum is 24 hours.

ThingsBoard stores the key and records its expiry in the device's `expirationTime` server attribute.

### 3. Customer Claims the Device

The customer uses the **Claim Device** widget (or the REST API below), entering the device name and the secret key shown on the device.

## Server-Side Claiming

The tenant admin pre-generates secret keys and uploads them as device attributes. Keys reach customers via email, product packaging, or a portal — no device involvement required. This is the right choice for LoRaWAN/NB-IoT devices that are auto-provisioned via <DocLink product={props.product} path='TODO'>Platform Integrations</DocLink> and have no display.

### 1. Upload the Claiming Data

For each device, set a server-side attribute named `claimingData`:

```json
{
  "secretKey": "your-secret",
  "expirationTime": 1735689600000
}
```

- `secretKey` — the secret to distribute to the customer.
- `expirationTime` — Unix timestamp in milliseconds after which claiming is no longer allowed.

This can be done via the UI (**Device → Attributes → Add**) or in bulk via the <DocLink product={props.product} path='TODO'>REST API</DocLink>.

### 2. Distribute the Key

Send the device name and secret key to the customer through any external channel. The customer will need both to complete claiming.

### 3. Customer Claims the Device

Same as device-side: the customer uses the **Claim Device** widget or calls the REST API.

## Claiming via REST API

Customers can claim a device programmatically:

```http
POST /api/customer/device/{deviceName}/claim
Content-Type: application/json

{"secretKey": "your-secret"}
```

On success the device is assigned to the requesting customer's account.

## Reclaiming a Device

To return a device to the tenant (remove customer ownership):

```http
DELETE /api/customer/device/{deviceName}/claim
```

After reclaiming, the `claimingAllowed` attribute is removed (when `SECURITY_CLAIM_ALLOW_CLAIMING_BY_DEFAULT` is `false`), so the device cannot be claimed again until it is re-enabled.

## Configuration Reference

| Environment Variable | Default | Description |
|----------------------|---------|-------------|
| `SECURITY_CLAIM_ALLOW_CLAIMING_BY_DEFAULT` | `true` | Allow all devices to be claimed without a per-device `claimingAllowed` attribute |
| `SECURITY_CLAIM_DURATION` | `86400000` | Default claiming key validity in milliseconds (24 hours) |
| `CACHE_SPECS_CLAIM_DEVICES_TTL` | `1440` | Maximum time-to-live for the in-memory claim cache (minutes) |
