This page assumes you already have a public, browser-reachable HTTPS stream URL. If your camera only speaks RTSP on your LAN, <DocLink product={props.product} path="reference/widgets/video/deploy">build a public stream URL</DocLink> first, then come back here.

The steps below apply to both protocols. Whether you brought your own URL or built one for your camera, only the URL string changes.

## Create a camera device and the stream URL attribute

Each camera is represented by a device entity. The widget reads the stream URL from a server attribute on that device. **The attribute name is up to you** — the only requirement is that the widget's data key matches it. The examples below use `hlsStreamUrl` and `webrtcStreamUrl` because those are the widget defaults, but any name works (`mainCameraUrl`, `entrance_cam`, …).

<Steps>
1. Create a device for your camera under <DocLink product={props.product} path="user-guide/devices">Devices</DocLink>. The device profile and label are not significant to this widget — any profile works.
2. Open the device's **Attributes** tab → **Server attributes** → **+ Add attribute**.
3. Set:
   - **Key:** any name you want (defaults `hlsStreamUrl` / `webrtcStreamUrl`).
   - **Value type:** `String`.
   - **Value:** the public URL — for example, `https://cameras.yourdomain.com/camera1/index.m3u8` or `https://cameras.yourdomain.com/camera1/whep`.

## HLS Video Stream — settings reference

Open the widget settings and use the table below as a reference for each field.

<MultiProductImageGallery product={props.product} images={[
  {
    src: '/src/assets/images/reference/widgets/video-streaming/hls-widget-settings.png',
    alt: 'HLS Video Stream widget edit panel on the Appearance tab. The Video streaming section shows the Fallback stream URL field, Autoplay, Start muted, Show video controls toggles, and the Back buffer length field set to 10 seconds. The dashboard preview on the left plays a live HLS feed.',
    caption: 'HLS Video Stream widget settings — Appearance tab.',
  },
]} />

| Setting | Default | Description |
|---|---|---|
| **Fallback&nbsp;stream&nbsp;URL** | *(empty)* | Optional fallback URL. Used when the data key is empty or missing. Useful when you want a "default camera" to display before the user selects a device. |
| **Autoplay** | On | Start playback automatically when the stream loads. If the browser blocks autoplay with sound, the widget falls back to muted autoplay. |
| **Start&nbsp;muted** | On | Start playback muted. Users can unmute via the video controls. Required for reliable autoplay in most browsers. |
| **Show&nbsp;video&nbsp;controls** | On | Show the standard HTML5 video controls (play / pause, volume, fullscreen, picture-in-picture). |
| **Back&nbsp;buffer&nbsp;length** | 10 sec | Seconds of already-played live video kept in memory for seeking back. Valid range: 0–60. Set to 0 to disable seek-back. |

The widget supports **one entity, one data key**. The default data key label is `hlsStreamUrl`, but you can rename it to match any attribute name you've created on the device. By default, it ships with a `function` datasource pointing at a public Mux test stream (`https://test-streams.mux.dev/x36xhzz/x36xhzz.m3u8`) so it previews correctly; configure an Entity datasource bound to your device attribute.

The widget supports **one entity, one data key**. The default data key label is `webrtcStreamUrl`, but you can rename it to match any attribute name you've created on the device.

## Troubleshooting

**Widget displays "Can't reach the stream. Check the URL and try again."**
The widget's loading watchdog fired after 10 seconds without a successful start. Open browser DevTools → Network tab, reload the dashboard, and look at the request the widget made:

- A failed CORS preflight or `Access-Control-Allow-Origin` mismatch means the stream server isn't allowing your dashboard's origin. If you control the stream server, configure it to send `Access-Control-Allow-Origin: *` (or the specific dashboard origin) on stream responses. If you set this up yourself, see the matching item under <DocLink product={props.product} path="reference/widgets/video/deploy">Build a public stream URL</DocLink> → Troubleshooting.
- A `404` on `index.m3u8` or `whep` means the URL's path doesn't exist on the stream server. Double-check the path with whoever manages the stream. If you followed <DocLink product={props.product} path="reference/widgets/video/deploy">Build a public stream URL</DocLink>, the path segment must match an `MTX_PATHS_*` entry in your `docker-compose.yml`.

**Mixed-content blocking.** If the dashboard is served over HTTPS but the stream URL is `http://`, the browser blocks the request silently. The widget then shows the "no stream" placeholder. Use an HTTPS URL.
