The <a href="https://www.unified-automation.com/downloads/opc-ua-servers.html" target="_blank"><strong>OPC UA C++ Demo Server</strong></a> exposes a `BuildingAutomation` object with 10 simulated **AirConditioner** devices, each reporting `Temperature`, `Humidity`, `State`, and `PowerConsumption` values. The ThingsBoard OPC-UA integration subscribes to these device nodes, decodes their values into telemetry via an uplink converter, and — through a downlink converter and a dedicated rule chain — turns any airconditioner on or off by calling its `Start` / `Stop` OPC-UA method, all from a live dashboard.

This recipe walks through one concrete scenario end to end. For a full field-by-field reference of every OPC-UA integration setting, see <DocLink product={props.product} path="user-guide/integrations/opc-ua" target="_blank">OPC-UA Integration</DocLink>.

<Aside type="note" title="OPC UA vs. OPC-UA">
"OPC UA" (no hyphen) is the protocol and vendor product name, as in **OPC UA C++ Demo Server**. "OPC-UA" (hyphenated) is ThingsBoard's spelling for the integration type. Both refer to the same protocol.
</Aside>

## Prerequisites

- A <DocLink product={Products.PE} path="installation/" target="_blank">ThingsBoard Professional Edition</DocLink> instance installed locally;
- The <a href="https://www.unified-automation.com/downloads/opc-ua-servers.html" target="_blank"><strong>OPC UA C++ Demo Server</strong></a> (Windows only).

<Aside type="tip" title="OPC-UA Addressing Basics">
Every OPC-UA node has a **NodeId** built from a **namespace index** (which OPC-UA namespace the node belongs to) and an **identifier** (the node's name within that namespace, e.g. `AirConditioner_1`). The **Fully Qualified Name (FQN)** is the full browsed path to a node, e.g. `Objects.BuildingAutomation.AirConditioner_1`. This recipe uses all three forms: the uplink converter receives `opcUaNode_namespaceIndex` / `opcUaNode_identifier` / `opcUaNode_fqn` as metadata (Step 2), and the downlink converter builds NodeIds manually for write and method calls (Step 3).
</Aside>

## Step 2. Create the Uplink Converter

The integration wizard (Step 4) requires selecting an existing uplink and downlink converter, so both are created first.

The uplink converter decodes incoming OPC-UA node values and maps them to the ThingsBoard data model. The decoder function receives:
- `payload` — JSON object with node values, e.g. `{"temperature": "72.15819999999641"}`
- `metadata` — node metadata: `opcUaNode_name`, `opcUaNode_namespaceIndex`, `opcUaNode_identifier`, `opcUaNode_fqn`, `integrationName`

**Sample payload:**
```json
{
    "temperature": "72.15819999999641"
}
```

**Sample metadata:**
```json
{
    "opcUaNode_namespaceIndex": "3",
    "opcUaNode_name": "AirConditioner_1",
    "integrationName": "OPC-UA Airconditioners",
    "opcUaNode_identifier": "AirConditioner_1",
    "opcUaNode_fqn": "Objects.BuildingAutomation.AirConditioner_1"
}
```

For the full decoder function reference, see <DocLink product={props.product} path="user-guide/integrations/uplink-data-converter" target="_blank">Uplink data converter</DocLink>.

## Step 3. Create the Downlink Converter

The downlink converter encodes RPC commands from ThingsBoard into OPC-UA write operations or method calls. The encoder must return a single object with these fields:
- `contentType` — `JSON`, `TEXT`, or `BINARY`
- `data` — JSON string with two optional arrays:
  - `writeValues` — `{nodeId, value}` pairs; NodeId format: `ns=<namespaceIndex>;<type>=<identifier>` where `<type>` is `s` (string), `i` (numeric), `g` (GUID), or `b` (byte string)
  - `callMethods` — `{objectId, methodId, args}` objects for calling OPC-UA methods
- `metadata` — must be an empty object (`{}`) for OPC-UA

The output of the downlink converter must follow this structure:

```json
[{
    "contentType": "JSON",
    "data": "{\"writeValues\":[],\"callMethods\":[{\"objectId\":\"ns=3;s=AirConditioner_1\",\"methodId\":\"ns=3;s=AirConditioner_1.Stop\",\"args\":[]}]}",
    "metadata": {}
}]
```

For the full encoder function reference, see <DocLink product={props.product} path="user-guide/integrations/downlink-data-converter" target="_blank">Downlink data converter</DocLink>.

<Steps>
  1. Go to **Integrations center &#8702; Data converters**.
  2. Click **+ Add data converter &#8702; Create new converter**.
  3. Set **Converter type** to **Downlink**.
  4. Select **integration type** from the dropdown — **OPC-UA**.
  5. Enter a converter **name**: `OPC-UA Downlink Converter`
  6. In the **Main encoding configuration**, enter an encoder function:
</Steps>

```js maxLines=15 collapsible
var data = {
    writeValues: [],
    callMethods: []
};

if (msgType === 'RPC_CALL_FROM_SERVER_TO_DEVICE') {
    if (msg.method === 'setState') {
        var targetMethod = msg.params === 'true' ? 'Start' : 'Stop';
        var writeValue = {
              nodeId: 'ns=' + metadata['cs_namespaceIndex'] +';s=' + metadata['deviceName'],
              value: msg.params
        };

              objectId: 'ns=' + metadata['cs_namespaceIndex'] +';s=' + metadata['deviceName'],
              methodId: 'ns=' + metadata['cs_namespaceIndex'] +';s=' + metadata['deviceName']+'.'+targetMethod,
              args: []
        };
        data.callMethods.push(callMethod);
    }
}

var result = {
    contentType: "JSON",
    data: JSON.stringify(data),
    metadata: {}
};

return result;
```

<Aside type="caution" title="Where cs_namespaceIndex Comes From">
`metadata['cs_namespaceIndex']` isn't set by anything in this recipe so far — it's populated later, in Step 6, by a **Get Attributes** node embedded in the imported Airconditioners rule chain. That node reads the `namespaceIndex` client-scope attribute — the same value the uplink converter wrote as `result.attributes.namespaceIndex` in Step 2 — and injects it into the message metadata; ThingsBoard prefixes client-scope attributes with `cs_` when they're added to metadata this way. This means downlink control only works for a device **after** it has sent at least one uplink message — see <a href="#troubleshooting">Troubleshooting</a> if a switch appears to do nothing.
</Aside>

## How It Works

On the uplink path, the OPC-UA integration polls the subscribed device nodes, matches them against the **Device Node Pattern**, and passes each tag's raw value to the uplink converter's `decodeToJson(payload)`. The decoder maps `temperature`, `humidity`, and `powerConsumption` to telemetry and `state` to a device attribute.

On the downlink path, toggling the **Round switch** sends an `RPC_CALL_FROM_SERVER_TO_DEVICE` message with `method: setState` and `params` of `"true"`/`"false"`. The **Airconditioners** rule chain, wired to the **RPC Request to Device** relation on the Root Rule Chain, first runs a **Get Attributes** node that reads the `namespaceIndex` client-scope attribute and injects it into the message metadata as `cs_namespaceIndex` (see <a href="#step-3-create-the-downlink-converter"><strong>Step 3</strong></a>), then routes the message to the **integration downlink** node. The downlink converter's encoder builds the OPC-UA `nodeId` from `metadata['cs_namespaceIndex']` and `metadata['deviceName']`, and calls the `Start` or `Stop` method on that node depending on the requested state.

## Troubleshooting

**No devices appear after adding the integration** — confirm `UaCPPServer` is still running and that **Host**/**Port** match the values shown in the UA Admin Dialog (<a href="#step-1-launch-the-opc-ua-demo-server"><strong>Step 1</strong></a>). A connection failure otherwise fails silently — no devices will appear on the Devices page.

**Devices appear but telemetry never updates** — check that the **Subscription tags** paths (`Temperature`, `Humidity`, `State`, `PowerConsumption`) exactly match the child node names on the server; these are case-sensitive.

**Fewer or no devices are created** — the **Device Node Pattern** regex doesn't match the server's actual node FQNs. Browse the server's node tree to confirm the exact path, then adjust the pattern.

**The Round switch has no effect** — this is expected for a device that hasn't sent an uplink message yet: the OPC-UA `nodeId` used for the `Start`/`Stop` call depends on the `namespaceIndex` client attribute, which is only set once the uplink converter has processed at least one message for that device (see <a href="#step-3-create-the-downlink-converter"><strong>Step 3</strong></a>). Wait for telemetry to appear (<a href="#step-5-verify-uplink-telemetry"><strong>Step 5</strong></a>) before testing downlink control.

**Connecting to a real (non-demo) server fails** — a production OPC-UA server typically enforces a **Security** policy other than `None` and requires **Identity** credentials rather than `Anonymous`; see <DocLink product={props.product} path="user-guide/integrations/opc-ua/#connection-settings" target="_blank">Connection settings</DocLink>.
