Welcome to **ThingsBoard IoT Hub**. This guide walks you through contributing a calculated field that end users can install on their ThingsBoard instance in a single click.

A calculated field contribution is a **single JSON file** that you export from ThingsBoard and upload through IoT Hub. When a user clicks **Install** on your calculated field, the platform imports it into their device or asset profile where it starts computing derived values from live telemetry.

## How Users Install Your Calculated Field

When a user installs your calculated field from IoT Hub:

1. The platform downloads your calculated field JSON file.
2. The user picks a target device or asset profile and imports the calculated field into it.
3. The calculated field immediately starts processing live telemetry — every incoming message runs through your formula or script and the computed result is saved back as a new telemetry key or attribute.

A calculated field contribution is self-contained — the formula, script, or rule is entirely in the `configuration` object. End users do not need to install dependencies or write code.

## Quick Start

1. In your ThingsBoard instance, create and test your calculated field on a device or asset profile.
2. Export it as a JSON file — see [Exporting a Calculated Field from ThingsBoard](#exporting-a-calculated-field-from-thingsboard).
3. Go to **IoT Hub → Add Item → Calculated Field**. A four-step wizard walks you through the rest:
    - **Type & Data** — upload your exported JSON. The platform parses it and pre-fills the Name from the file.
    - **Listing** — fill in the listing metadata (see [The Listing Step](#the-listing-step)). Name is pre-filled; Icon, Color, Description, Categories, Use Cases, and the supported ThingsBoard version must be selected manually.
    - **Readme** — write the long-form documentation in Markdown (see [Readme Content](#readme-content)).
    - **Review & Submit** — verify everything and submit the version for review.

The sections below are a complete reference for each step.

## Exporting a Calculated Field from ThingsBoard

1. Open a device or asset profile in ThingsBoard.
2. Go to the **Calculated fields** tab and open the field you want to export.
3. Click the **Export calculated field** button (download icon in the top toolbar).
4. Save the resulting `.json` file — this is your upload file.

The exported file is a complete calculated field definition and requires no further editing to be valid.

## Calculated Field JSON Structure

A valid calculated field file is a JSON object with three required fields — `name`, `type`, and `configuration` — plus optional metadata.

```json
{
  "name": "Dew Point",
  "type": "SIMPLE",
  "debugSettings": {
    "failuresEnabled": true,
    "allEnabled": false
  },
  "configuration": {
    "type": "SIMPLE",
    "arguments": {
      "t": {
        "refEntityKey": {
          "key": "temperature",
          "type": "TIME_SERIES",
          "scope": null
        },
        "defaultValue": null
      },
      "h": {
        "refEntityKey": {
          "key": "humidity",
          "type": "TIME_SERIES",
          "scope": null
        },
        "defaultValue": null
      }
    },
    "output": {
      "type": "TIME_SERIES",
      "name": "dewPoint",
      "scope": null,
      "decimalsByDefault": 2
    },
    "expression": "t - ((100 - h) / 5)"
  }
}
```

### Required Fields

| Field | Type | Description |
|-------|------|-------------|
| `name` | string | Non-empty calculated field name (e.g. `Dew Point`) |
| `type` | string | One of the [Calculated Field Types](#calculated-field-types) |
| `configuration` | object | Type-specific configuration — must include `arguments`, `output`, and either `expression` (SIMPLE) or `script` (SCRIPT) |

### Configuration Structure

The `configuration` object varies by calculated field type. Common fields across most types:

| Field | Description |
|-------|-------------|
| `arguments` | Map of argument name → reference (telemetry key, attribute key, or related entity property) the formula reads from |
| `output` | Where the result is saved — `type` (`TIME_SERIES` or `ATTRIBUTE`), `name` (key to write), `scope` (for attributes only) |
| `expression` | For `SIMPLE` type — a math expression referencing argument names |
| `script` | For `SCRIPT` type — a JavaScript function body that returns the computed value |

See the [ThingsBoard Calculated Fields documentation](https://thingsboard.io/docs/user-guide/calculated-fields/) for full per-type configuration schemas.

## The Listing Step

The Listing step of the upload wizard collects the fields shown on the browse card and detail page. Only the Name is pre-filled from the JSON; the rest are manual.

| Field | Required | Source | Notes |
|-------|:--------:|--------|-------|
| Name | yes | `name` in JSON | Editable |
| Icon | yes | Manual | Material Icon — see [Icon and Color](#icon-and-color) |
| Color | yes | Manual | Hex color for the icon background |
| Description | yes | Manual | One-sentence summary, max 512 chars |
| Categories | yes | Manual | Pick from [Calculated Field Categories](#calculated-field-categories) |
| Use Cases | yes | Manual | Pick from [Use Cases](#use-cases) |
| Supported ThingsBoard version | no | Manual | Min (`≥`) and optional max (`<`) version |
| Professional Edition | no | Manual | Toggle if the calculated field requires PE |
| Tags | no | Manual | Free-form keywords |

### You Provide via the Listing Form

| Field | Purpose |
|-------|---------|

Long-form documentation — what the field computes, input keys, output key, formula, installation target, and edge cases. See [Readme Content](#readme-content).

### Set Per Version

Version (semver) and changelog — bumped every time you upload a changed JSON.

## Allowed Values

### Calculated Field Types

The `type` field must be one of: `SIMPLE`, `SCRIPT`, `GEOFENCING`, `ALARM`, `PROPAGATION`, `RELATED_ENTITIES_AGGREGATION`, `ENTITY_AGGREGATION`.

| Type | Use for |
|------|---------|
| `SIMPLE` | Math expressions over telemetry/attribute values (e.g. unit conversion, derived metrics like dew point or enthalpy) |
| `SCRIPT` | Custom JavaScript logic — for formulas too complex for an expression or requiring branching |
| `GEOFENCING` | Detecting entry/exit of geographical zones based on latitude/longitude telemetry |
| `ALARM` | Alarm-driven fields — values computed from the entity's alarm state |
| `PROPAGATION` | Propagating values up or down the entity hierarchy (e.g. from devices to assets) |
| `RELATED_ENTITIES_AGGREGATION` | Aggregating telemetry across related entities (sum, avg, min, max across devices linked to an asset) |
| `ENTITY_AGGREGATION` | Aggregating telemetry across entities of a given profile (fleet-wide averages, totals) |

### Calculated Field Categories

Pick one or more categories that describe what your calculated field does. Allowed values: `Aggregation`, `Custom Formula`, `Energy`, `Environmental`, `Geospatial`, `Predictive`, `Statistical`.

### Use Cases

Pick one or more IoT domains where your calculated field is useful. These are shared across the whole marketplace.

Allowed values: `Air Quality Monitoring`, `Asset Tracking`, `Cold Chain`, `Drones`, `Environment Monitoring`, `Fleet Tracking`, `Health Care`, `Industrial Automation`, `Predictive Maintenance`, `Robotics`, `SCADA`, `Smart Building`, `Smart City`, `Smart Energy`, `Smart Farming`, `Smart Home`, `Smart Metering`, `Smart Office`, `Smart Retail`, `Solar Monitoring`, `Tank Level Monitoring`, `Waste Management`.

## Icon and Color

Calculated fields are displayed in the catalog with a **Material Icon** on a **colored background** instead of a preview image. Both are required in the Listing step.

### Icon

Pick a Material Icon that visually represents the field's purpose. The icon picker shows a curated list of icons relevant to calculated fields (e.g. `calculate`, `functions`, `analytics`, `trending_up`). You can also search the full Material Icons set.

**Good icon choices:**

- Math expressions → `calculate`, `functions`, `pi`
- Script-based → `code`, `terminal`
- Geospatial → `location_on`, `my_location`, `place`
- Alarm-driven → `notifications_active`, `warning`
- Propagation → `trending_up`, `merge_type`
- Aggregation → `aggregate`, `summarize`, `analytics`

### Color

Pick a hex color for the icon background. The picker provides 24 presets and a custom color picker.

**Conventions across the catalog:**

- Green tones for environmental and energy calculations
- Blue tones for geospatial
- Orange tones for alarm and alerting-related fields
- Purple for predictive and machine-learning-based fields

## Readme Content

The Readme step of the upload wizard collects the long-form documentation shown on your calculated field's detail page. It is written in standard Markdown.

A good readme always includes:

1. **What the field computes** — one or two sentences with the underlying formula or algorithm
2. **Inputs** — the exact telemetry key names or attribute names the field reads from, with expected units
3. **Output** — the key the field writes to and its unit
4. **Formula or algorithm** — the math or logic, explained so users can verify correctness
5. **Installation target** — device profile, asset profile, or either
6. **Assumptions and edge cases** — range limits, behavior when inputs are missing or out of range

Example:

````markdown
## Dew Point

Calculates the approximate dew point temperature (°C) from ambient temperature and relative humidity telemetry using the simplified Magnus formula.

### Inputs

| Key | Type | Unit | Notes |
|-----|------|------|-------|
| `temperature` | timeseries | °C | Ambient temperature |
| `humidity` | timeseries | % | Relative humidity, 0–100 |

### Output

| Key | Type | Unit |
|-----|------|------|
| `dewPoint` | timeseries | °C |

### Formula

```
dewPoint = temperature - ((100 - humidity) / 5)
```

This simplified formula is accurate to within ±1°C when humidity is above 50%.

### Installation Target

Any device profile that publishes `temperature` and `humidity` telemetry.

### Edge Cases

- If either input is missing, no output is produced
- For humidity below 50%, consider using a `SCRIPT` type field with the full Magnus-Tetens formula
````

## Versioning & Checksum

Each calculated field version is identified by a semver string (e.g. `1.0.0`) and a SHA-256 **checksum** computed from:

- `name`
- `type`
- `configuration` (the entire configuration object as JSON)

This means any change to the formula, script, arguments, or output configuration changes the checksum and requires bumping the version. Changes to description, icon, or color do not change the checksum.

**Always bump the version** when uploading a changed calculated field JSON.

## Pre-Upload Checklist

### Calculated Field JSON

- [ ] `name` is non-empty and descriptive
- [ ] `type` is one of `SIMPLE`, `SCRIPT`, `GEOFENCING`, `ALARM`, `PROPAGATION`, `RELATED_ENTITIES_AGGREGATION`, `ENTITY_AGGREGATION`
- [ ] `configuration` is a valid object for the declared `type`
- [ ] All `arguments` reference existing telemetry or attribute keys
- [ ] `output.name` does not clash with common key names that the user might already use (e.g. prefer `dewPoint` over `dp` or `value`)
- [ ] For `SIMPLE` type: `expression` is syntactically valid and uses only declared arguments
- [ ] For `SCRIPT` type: `script` is valid JavaScript and returns a numeric or string value

### Listing Form

- [ ] Icon is set to a Material Icon name
- [ ] Color is set to a hex value (e.g. `#2196F3`)
- [ ] Description is one sentence, under 512 characters
- [ ] At least one value selected from the [Calculated Field Categories](#calculated-field-categories) list
- [ ] At least one value selected from the [Use Cases](#use-cases) list
- [ ] Minimum ThingsBoard version is set appropriately
- [ ] Professional Edition toggle matches whether the field uses PE-only features

### Readme

- [ ] Documents the exact input key names (lowercase, units, expected ranges)
- [ ] Documents the output key and its unit
- [ ] Explains the formula or algorithm
- [ ] States the intended installation target (device profile, asset profile, or either)
- [ ] Covers edge cases (missing inputs, out-of-range values)

### Content Quality

- [ ] Calculated field has been tested on a real ThingsBoard instance with realistic telemetry
- [ ] The output key name does not collide with common device telemetry keys
- [ ] Icon and color are consistent with the catalog conventions for your category
- [ ] Version follows semver (e.g. `1.0.0`)
