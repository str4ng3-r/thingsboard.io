Welcome to **ThingsBoard IoT Hub**. This guide walks you through contributing a rule chain that end users can install on their ThingsBoard instance in a single click.

A rule chain contribution is a **single JSON file** that you export from ThingsBoard and upload through IoT Hub. When a user clicks **Install** on your rule chain, the platform imports it directly where they can link it into their message flow.

## How Users Install Your Rule Chain

When a user installs your rule chain from IoT Hub:

1. The platform downloads your rule chain JSON file.
2. The rule chain is imported into the user's **Rule Chains** list.
3. The user can wire it into their root rule chain, set it as the default for a device profile, or deploy it to edge instances (for `EDGE` type chains).

A rule chain contribution is self-contained — every node (its type, name, configuration) and every connection between nodes are encoded in the JSON. End users do not need to install any dependencies or configure anything outside ThingsBoard.

## Quick Start

1. In your ThingsBoard instance, build and test your rule chain in the Rule Chains editor.
2. Export it as a JSON file — see [Exporting a Rule Chain from ThingsBoard](#exporting-a-rule-chain-from-thingsboard).
3. Go to **IoT Hub → Add Item → Rule Chain**. A four-step wizard walks you through the rest:
    - **Type & Data** — upload your exported JSON. The platform parses it and pre-fills the Name from the file.
    - **Listing** — fill in the listing metadata (see [The Listing Step](#the-listing-step)). Name is pre-filled; Icon, Color, Description, Categories, Use Cases, and the supported ThingsBoard version must be selected manually.
    - **Readme** — write the long-form documentation in Markdown (see [Readme Content](#readme-content)).
    - **Review & Submit** — verify everything and submit the version for review.

The sections below are a complete reference for each step.

## Exporting a Rule Chain from ThingsBoard

1. Open **Rule Chains** in your ThingsBoard instance.
2. Find your rule chain in the list.
3. Click the **Export rule chain** button (download icon on the row).
4. Save the resulting `.json` file — this is your upload file.

The exported file is a complete rule chain definition and requires no further editing to be valid.

## Rule Chain JSON Structure

A valid rule chain file is a JSON object with a `ruleChain` header, a `metadata` body containing `nodes`, and optional `connections`.

```json
{
  "ruleChain": {
    "name": "Temperature Alarm Chain",
    "type": "CORE",
    "root": false,
    "debugMode": false,
    "configuration": null,
    "additionalInfo": {
      "description": "Triggers alarms when temperature exceeds thresholds"
    }
  },
  "metadata": {
    "firstNodeIndex": 0,
    "nodes": [
      {
        "type": "org.thingsboard.rule.engine.filter.TbMsgTypeFilterNode",
        "name": "Telemetry",
        "debugMode": false,
        "configuration": {
          "messageTypes": ["POST_TELEMETRY_REQUEST"]
        },
        "additionalInfo": {
          "layoutX": 200,
          "layoutY": 200
        }
      },
      {
        "type": "org.thingsboard.rule.engine.action.TbCreateAlarmNode",
        "name": "Create High Temp Alarm",
        "debugMode": false,
        "configuration": {
          "alarmType": "High Temperature",
          "severity": "WARNING",
          "propagate": true
        }
      }
    ],
    "connections": [
      {
        "fromIndex": 0,
        "toIndex": 1,
        "type": "True"
      }
    ],
    "ruleChainConnections": []
  }
}
```

### Required Structure

| Path | Type | Description |
|------|------|-------------|
| `ruleChain` | object | Rule chain header — must be present |
| `ruleChain.name` | string | Non-empty rule chain name |
| `ruleChain.type` | string | One of the [Rule Chain Types](#rule-chain-types) |
| `metadata` | object | Rule chain body — must be present |
| `metadata.nodes` | array | At least an empty array — typically contains your filter, transformation, and action nodes |

### Node Structure

Each entry in `metadata.nodes` represents one rule node on the canvas:

| Field | Description |
|-------|-------------|
| `type` | Fully qualified Java class name of the node (e.g. `org.thingsboard.rule.engine.filter.TbMsgTypeFilterNode`) |
| `name` | Display name shown on the canvas |
| `debugMode` | Whether debug logging is enabled for this node |
| `configuration` | Node-specific configuration object |
| `additionalInfo` | Editor metadata (position on canvas, etc.) |

### Connection Structure

Each entry in `metadata.connections` wires two nodes together:

| Field | Description |
|-------|-------------|
| `fromIndex` | Index of the source node in the `nodes` array |
| `toIndex` | Index of the target node in the `nodes` array |
| `type` | Connection label (e.g. `Success`, `Failure`, `True`, `False`) — depends on the source node's output ports |

## The Listing Step

The Listing step of the upload wizard collects the fields shown on the browse card and detail page. Only the Name is pre-filled from the JSON; the rest are manual.

| Field | Required | Source | Notes |
|-------|:--------:|--------|-------|
| Name | yes | `ruleChain.name` in JSON | Editable |
| Icon | yes | Manual | Material Icon — see [Icon and Color](#icon-and-color) |
| Color | yes | Manual | Hex color for the icon background |
| Description | yes | Manual | One-sentence summary, max 512 chars |
| Categories | yes | Manual | Pick from [Rule Chain Categories](#rule-chain-categories) |
| Use Cases | yes | Manual | Pick from [Use Cases](#use-cases) |
| Supported ThingsBoard version | no | Manual | Min (`≥`) and optional max (`<`) version |
| Professional Edition | no | Manual | Toggle if the rule chain requires PE |
| Tags | no | Manual | Free-form keywords |

## What's Auto-Derived vs What You Provide

### Auto-Derived from Rule Chain JSON

| Listing Field | Source |
|-------------------|--------|
| Name | `ruleChain.name` |
| Rule chain type | `ruleChain.type` |
| Node count | Count of entries in `metadata.nodes` |
| Node summary | Deduplicated node type categories (filter, transformation, action, etc.) |

### You Provide via the Listing Form

| Field | Purpose |
|-------|---------|
| Icon | Material icon shown on browse cards |
| Color | Hex background color for the icon |
| Description | One-sentence summary shown on browse cards |
| Categories | Browse filtering |
| Use Cases | Browse filtering |
| Supported ThingsBoard version | Compatibility constraints |
| Professional Edition | Edition targeting |
| Tags | Free-form keywords used for search |

### You Provide in the Readme Step

Long-form documentation — what the rule chain does, trigger, behavior, configuration, outputs, and wiring instructions. See [Readme Content](#readme-content).

### Set Per Version

Version (semver) and changelog — bumped every time you upload a changed JSON.

## Allowed Values

### Rule Chain Types

The `ruleChain.type` field must be one of: `CORE`, `EDGE`.

| Type | Use for |
|------|---------|
| `CORE` | A rule chain that runs on the main ThingsBoard server — the common case |
| `EDGE` | A rule chain that runs on a ThingsBoard Edge instance — for edge message routing |

### Rule Chain Categories

Pick one or more categories that describe what your rule chain does. Allowed values: `Alerting`, `Analytics`, `Data Processing`, `Device Connectivity`, `Integration`.

### Use Cases

Pick one or more IoT domains where your rule chain is useful. These are shared across the whole marketplace.

Allowed values: `Air Quality Monitoring`, `Asset Tracking`, `Cold Chain`, `Drones`, `Environment Monitoring`, `Fleet Tracking`, `Health Care`, `Industrial Automation`, `Predictive Maintenance`, `Robotics`, `SCADA`, `Smart Building`, `Smart City`, `Smart Energy`, `Smart Farming`, `Smart Home`, `Smart Metering`, `Smart Office`, `Smart Retail`, `Solar Monitoring`, `Tank Level Monitoring`, `Waste Management`.

## Icon and Color

Rule chains are displayed in the catalog with a **Material Icon** on a **colored background** instead of a preview image. Both are required in the Listing step.

### Icon

Pick a Material Icon that visually represents what the rule chain does. The icon picker shows a curated list of icons relevant to rule chains (e.g. `account_tree`, `device_hub`, `hub`, `filter_alt`, `notifications_active`). You can also search the full Material Icons set.

**Good icon choices:**

- Alerting chains → `notifications_active`, `warning`, `priority_high`
- Analytics chains → `analytics`, `insights`, `query_stats`
- Data processing chains → `filter_alt`, `transform`, `timeline`
- Device connectivity chains → `device_hub`, `hub`, `lan`
- Integration chains → `integration_instructions`, `sync_alt`

### Color

Pick a hex color for the icon background. The picker provides 24 presets and a custom color picker.

**Conventions across the catalog:**

- Red tones for alerting
- Blue tones for integration and connectivity
- Green tones for transformations and analytics
- Purple for machine-learning or custom-logic chains

### Why Icons Instead of Images?

Rule chains don't have a natural visual representation like widgets or dashboards do. A color-coded icon scales cleanly at any size on cards, headers, and dialog previews, and keeps the catalog layout consistent.

## Readme Content

The Readme step of the upload wizard collects the long-form documentation shown on your rule chain's detail page. It is written in standard Markdown.

A good readme always includes:

1. **What the rule chain does** — one or two sentences
2. **Trigger** — what message type(s) and source entity type(s) the chain operates on
3. **Behavior** — the logic flow in plain English, aligned with the node graph
4. **Configuration** — any fields inside nodes that users should tune for their data (e.g. thresholds, alarm severities, rate limits)
5. **Outputs** — what the chain produces (alarms, attribute updates, forwarded messages, external HTTP calls)
6. **Wiring guide** — how to hook the chain into the user's root rule chain

Example:

```markdown
## Temperature Alarm Chain

Triggers `High Temperature` alarms when incoming telemetry exceeds a configurable threshold.

### Trigger

Processes `POST_TELEMETRY_REQUEST` messages from any device.

### Behavior

1. **Filter** — only telemetry messages with a `temperature` key are processed
2. **Threshold check** — temperature > 70°C routes to the alarm branch
3. **Create alarm** — raises a `High Temperature` alarm with `WARNING` severity, propagating to related assets
4. **Clear alarm** — temperature ≤ 70°C clears any existing alarm for the device

### Configuration

Tune these node fields to match your deployment:

| Node | Field | Purpose |
|------|-------|---------|
| Threshold check | `jsScript` | Change the threshold value (default 70) |
| Create alarm | `severity` | `CRITICAL`, `MAJOR`, `MINOR`, or `WARNING` |
| Create alarm | `propagate` | Whether to propagate to related assets |

### How to Use

Link this chain from your root rule chain's `Post Telemetry` output, or set it as the default rule chain for a temperature-sensor device profile.
```

## Versioning & Checksum

Each rule chain version is identified by a semver string (e.g. `1.0.0`) and a SHA-256 **checksum** computed from:

- `ruleChain.name`
- `ruleChain.type`
- Every node's `type`, `name`, and `configuration`
- Every connection's `fromIndex`, `toIndex`, and `type`

This means any structural change — adding a node, changing a node's configuration, rewiring connections — changes the checksum and requires bumping the version. Changes to description, icon, or color do not change the checksum.

**Always bump the version** when uploading a changed rule chain JSON.

## Pre-Upload Checklist

### Rule Chain JSON

- [ ] `ruleChain` object is present with `name` and `type`
- [ ] `ruleChain.type` is one of `CORE` or `EDGE`
- [ ] `metadata.nodes` is an array (may be empty, but usually contains your logic)
- [ ] All node `configuration` fields are valid for their respective node types
- [ ] All `connections` reference valid node indices and correct labels for the source node's output ports

### Listing Form

- [ ] Icon is set to a Material Icon name
- [ ] Color is set to a hex value (e.g. `#4CAF50`)
- [ ] Description is one sentence, under 512 characters
- [ ] At least one value selected from the [Rule Chain Categories](#rule-chain-categories) list
- [ ] At least one value selected from the [Use Cases](#use-cases) list
- [ ] Minimum ThingsBoard version is set appropriately
- [ ] Professional Edition toggle matches whether the chain uses PE-only nodes

### Readme

- [ ] Documents the exact message types and keys the chain expects
- [ ] Describes the behavior in plain English, step by step
- [ ] Lists every node field users should tune
- [ ] Explains how to wire the chain into the root rule chain

### Content Quality

- [ ] Rule chain has been tested on a real ThingsBoard instance with realistic data
- [ ] Node names in the JSON are descriptive (not `node_1`, `Copy of X`, etc.)
- [ ] Icon and color are consistent with the catalog conventions for your category
- [ ] Version follows semver (e.g. `1.0.0`)
